# AURA Capacity Planning — Metrics, Thresholds & APIs

> Source of truth: `src/services/aiInsights.ts`, `src/config/environmentProfiles.ts`, `src/components/dashboard/InsightCardsGrid.tsx`, `src/components/BestPracticesWidget.tsx`
> Generated: 2026-05-28

In AURA, "Capacity Planning" lives in two complementary surfaces:

1. **AI Insights → Capacity Planning group** — a rule-based forecast that warns when the network is approaching density / channel-utilization / client-growth thresholds for the active **Environment Profile**.
2. **Best Practices Evaluation** — the Campus Controller's own configuration & operational checks (`/v1/bestpractices/evaluate`), surfaced verbatim with status badges.

Both feed the same operator workflow: "what should I tune before this network falls over?"

---

## 1. Environment Profiles (the threshold pack)

The active profile sets the thresholds against which capacity is judged. Defined in `src/config/environmentProfiles.ts`.

| Profile     | rfqiTarget | rfqiPoor | channelUtil % | noiseFloor dBm | **clientDensity** | latencyP95 ms | retryRate % | interferenceHigh |
|-------------|-----------:|---------:|---------------:|----------------:|-------------------:|---------------:|-------------:|------------------:|
| AI Baseline | 75 | 55 | 65 | −85 | **50** | 75 | 15 | 0.30 |
| Retail      | 80 | 60 | 60 | −85 | **50** | 50 | 15 | 0.25 |
| Warehouse   | 70 | 50 | 70 | −80 | **20** | 100 | 20 | 0.35 |
| Distribution| 65 | 45 | 75 | −80 | **30** | 75 | 25 | 0.40 |
| **HQ**      | 85 | 65 | 50 | −90 | **100** | 30 | 10 | 0.20 |
| Campus      | 75 | 55 | 65 | −85 | **75** | 50 | 15 | 0.30 |
| Custom      | 75 | 55 | 65 | −85 | **75** | 50 | 15 | 0.30 |

Default profile when none is set: **Campus** (`getEnvironmentProfile()` fallback).

### Metric evaluation buckets

`evaluateMetric(profile, metric, value)` returns `'good' | 'warning' | 'poor'` using these bands:

| Metric | Good | Warning | Poor |
|---|---|---|---|
| rfqi               | ≥ rfqiTarget | ≥ rfqiPoor | < rfqiPoor |
| channelUtilization | ≤ 80% of threshold | ≤ threshold | > threshold |
| noiseFloor (dBm)   | ≤ threshold − 5 | ≤ threshold | > threshold |
| latency            | ≤ 70% of P95    | ≤ P95       | > P95 |
| retryRate          | ≤ 70% of threshold | ≤ threshold | > threshold |
| interference       | ≤ 70% of threshold | ≤ threshold | > threshold |

## 2. Capacity Planning Insights (rule-based)

Generated by `generateInsights()` in `src/services/aiInsights.ts`. Two of the rules belong to the **Capacity Planning** group; both gate on the active profile's thresholds.

### 2.1 High Client Density (real-time)

| Field | Value |
|---|---|
| Trigger | `clientsPerAP > thresholds.clientDensity × 1.2` |
| Severity | `warning` (escalates to `critical` if growth-rate gates also fire) |
| Group | `capacity_planning` |
| Evidence | clients-per-AP, profile name, threshold |
| Action text | Review high-density areas / consider load balancing / add APs |

### 2.2 Capacity Planning Forecast (predictive)

| Field | Value |
|---|---|
| Trigger | `growthRate > 0.1` (10%/day) **AND** `clientsPerAP > 0.8 × clientDensity` |
| Severity | `warning` if days-to-capacity > 3, else `info`; surfaced as `warning` when < 3 days |
| Group | `capacity_planning` |
| Category | `predictive` |
| Confidence | 0.65 |
| Recurrence | 0.8 |
| Prediction window | computed: `(threshold − current) / (current × growth)` days |
| Prediction basis | "24-hour growth trend" |
| Evidence | Current load, threshold, daily growth %, projected clients/AP |
| Action text | Plan AP deployment, review high-density areas, enable load balancing |

The forecast requires `metrics.history.clientCount24hAgo` to be present; otherwise the rule is skipped.

### 2.3 Sibling rules that surface as capacity drivers

| Rule | Trigger | Group |
|---|---|---|
| Channel Utilization | `channelUtilization > thresholds.channelUtilizationPct` | network_health (relevant — high util is a capacity symptom) |
| Interference        | `interference > thresholds.interferenceHigh` | network_health |
| Retry Rate          | `retryRate > thresholds.retryRatePct` | network_health |
| APs Offline         | `offlinePercent > 0` (any) | predictive_maintenance |

These are not in the `capacity_planning` group but a capacity recommendation typically references them.

## 3. Best Practices Evaluation (controller-side)

`BestPracticesWidget` reads the Campus Controller's own config & operational checks. Each `BestPractice` row has:

```ts
interface BestPractice {
  id: string;
  criteria: string;             // e.g., "Channel reuse plan", "Min-rate set"
  detailedDescription: string;  // human-readable why
  causeBy: Array<{ type, name, id }>;  // affected entities (APs, services, etc.)
  status: 'Good' | 'Warning' | 'Error';
  type: 'Config' | 'Operational';
}
```

UI behavior:

- Three summary tiles (Good / Warning / Error) act as filter chips.
- On first load the widget **auto-selects** the highest-severity bucket (Error > Warning > All).
- Sort options: **By Severity** (default) or **By Name**.

## 4. Defaults

| Setting | Default | Notes |
|---|---|---|
| Active profile (no override) | **Campus** | `getEnvironmentProfile('CAMPUS')` |
| Density warning gate (insight) | `clientsPerAP > 1.2 × profile.clientDensity` | High-density rule |
| Forecast growth gate | `growthRate > 10% / day` | Capacity Planning Forecast |
| Forecast approaching-capacity gate | `clientsPerAP > 80% × profile.clientDensity` | "soon to exceed" |
| Forecast confidence | 0.65 | Used by rank-score calc |
| Forecast recurrence | 0.80 | Used by rank-score calc |
| Best Practices initial filter | Highest-severity bucket present | Auto-selected once after first load |
| Best Practices sort | `severity` | User-changeable to `name` |
| Best Practices error state | "Best Practices API not available" on 404 | Card renders warning header |

## 5. APIs Used

### Best Practices (live controller data)

| Endpoint | Method | Purpose | Caller |
|---|---|---|---|
| `/v1/bestpractices/evaluate` | GET | Returns `{ timestamp, conditions: BestPractice[] }` for the connected controller — drives every status badge, counter, and recommendation in the widget | Inline `apiService.makeAuthenticatedRequest()` in `BestPracticesWidget.loadBestPractices()` |

A 404 from this endpoint is reported as "Best Practices API not available" — not all controller builds expose it.

### Capacity Planning Insights (derived from live data)

The insights pipeline is **client-side rules** running on metrics already fetched for the dashboard. There is **no dedicated `/insights` endpoint**. Inputs come from:

| Metric | Source endpoint | Caller |
|---|---|---|
| `clientCount` (real-time) | `/v1/stations` | `apiService.getStations()` (api.ts:1361) |
| `apOnlineCount` / offline | `/v1/aps/query` | `apiService.getAccessPoints()` (api.ts:1101) |
| `channelUtilization` | `/v1/aps/query` columns + (planned) `/v1/aps/ifstats` | currently inferred; ifstats not yet wired |
| `rfqi`, `noiseFloor`, `interference`, `retryRate` | Derived by `useDashboardData` from station/AP fields | `useDashboardData.ts` |
| `history.clientCount24hAgo` | Sliding window kept by `sleDataCollectionService` (1-min poll of `/v1/stations`) | `src/services/sleDataCollection.ts` |
| Active environment profile | `localStorage` (per site/org) | `useOperationalContext` / persona config |

### Authentication

Standard bearer token through `apiService.makeAuthenticatedRequest()`. Multi-controller deployments add `X-Controller-URL`.
